## M19 — Chat semantic memory recall

Closes the M18 deferred item. M18 replayed only the most-RECENT turns (a fixed sliding window); a relevant turn earlier in a long session fell out of the window and the only fallback was the linear summary fold. M19 adds an opt-in **retrieval-over-history** mode that replays the most-RELEVANT prior turns via pgvector cosine over per-message embeddings. New pure helper in `lib/chat-memory.ts`: `assembleRecallWindow(allMessagesWithIds, relevantIds, cfg) ⇒ {window, overflow}` = (semantic top-K ∪ the last `recencyTail` messages), deduped, restored to chronological order, bounded by the SAME `CHAT_MEMORY_TOKEN_BUDGET` + `CHAT_MEMORY_MAX_TURNS` as recency mode (walks newest→oldest so the current question's immediate context is never dropped for an older relevant turn). Config gains `CHAT_MEMORY_SEMANTIC_RECALL` (switch, default off) / `CHAT_MEMORY_SEMANTIC_RECALL_K` (default 8, positive) / `CHAT_MEMORY_SEMANTIC_RECALL_RECENCY_TAIL` (default 2, ≥0). New `lib/chat-recall.ts` (DB + embedder glue): `embedAndStoreChatMessage` (upsert one row per message into `chat_message_embeddings`, `ON CONFLICT (message_id)`) + `semanticRecallMessageIds` (cosine `<=>` top-K scoped to tenant + session + the **current** `embedder_version`, stable secondary sort on `message_id`; returns `[]` for a non-positive k / empty query so the caller falls back). Schema: new `chat_message_embeddings` table in `setup-sql.ts` (`message_id` PK → `chat_messages` `ON DELETE CASCADE`, `session_id`, `tenant_id`, `embedding vector(dim)`, `embedder_version`, `updated_at`; ivfflat cosine index + tenant/session indexes; RLS + FORCE RLS + `tenant_isolation`) — idempotent `CREATE TABLE IF NOT EXISTS`, provisioned by `db setup`, not push. Wiring in `routes/chat.ts`: when the switch is on, after persisting each user + assistant message it embeds the (already-PHI-safe) content best-effort; on each turn it embeds the current question, recalls the top-K, and `assembleRecallWindow` builds the replayed window (any failure → fall back to the recency `selectWindow`). **Embedder:** the shared `getEmbedder()` (PhiGuard-wrapped) — the offline `FeatureHashEmbedder` by default, so it works credential-free with no cloud SDK. **Orthogonal to the M18 summary:** the rolling summary still folds the RECENCY overflow (monotonic `coveredCount` preserved); semantic recall only swaps the window passed to `toHistoryTurns`, so the two compose (summary = old dropped context; recall = the most-relevant specific turns). **PHI posture:** `chat_messages.content` is already PHI-safe and the embedder is PhiGuard-wrapped, so no raw PHI reaches the vector tier; **no new ledger events and no LLM call** (pure retrieval). **Default-safe:** switch off ⇒ byte-identical to M18 (no embedding writes, no recall query); a fresh single-turn session has an empty window regardless; the credential-free eval gate never runs persisted multi-turn chat ⇒ byte-identical. Tests: new `assembleRecallWindow` cases + config defaults/switch in `chat-memory.test.ts` (26 cases total) + new `chat-recall.test.ts` DB integration suite (store + idempotent upsert, similarity ranking with `FeatureHashEmbedder`, k/empty-query guard, `embedder_version` filter, tenant+session scoping). **Both originally-deferred items are now built:** the hybrid BM25 ∪ vector fuse over chat history (M21) and per-tenant LIST partitioning of `chat_message_embeddings` (M22).

## M30 — OpenSearch ILM/ISM hot→warm→cold tiering (opt-in)

Surfaces a previously operator-only item as in-app config: when the lexical search leg runs on OpenSearch (`SEARCH_PROVIDER=opensearch`), the redacted-only mirror index can auto-tier through hot→warm→cold→[delete] via an OpenSearch ISM (Index State Management) policy the app provisions, instead of leaving lifecycle entirely to a cluster operator.

- **Config + policy builder (`lib/cloud-search.ts`)** — `loadIlmConfigFromEnv(env)` parses the opt-in switch `OPENSEARCH_ILM_ENABLED` + `OPENSEARCH_ILM_HOT_MAX_AGE` / `WARM_MAX_AGE` / `DELETE_MIN_AGE` / `POLICY_NAME`; pure `buildIsmPolicy(cfg, indexPrefix)` emits the hot→warm→cold→[delete] state machine (transitions by `min_index_age`) plus an `ism_template` that auto-attaches the policy to matching new indices by index pattern.
- **Client seam** — `OpenSearchClient` gains an optional `ensureIlmPolicy(name, body)`; the native impl issues `transport.request` `PUT _plugins/_ism/policies/<name>`. The provider applies the policy once (guarded) when ILM is enabled, around `ensureIndex`.
- **Default-INERT:** `OPENSEARCH_ILM_ENABLED` unset ⇒ no config parsed, no ISM request, search path identical; the whole OpenSearch path is itself inert unless `SEARCH_PROVIDER=opensearch`, so dev + the credential-free eval gate are byte-identical (+0.0pt). Tests: config parsing, ISM policy-doc shape, and provider-applies-once-when-enabled / never-when-disabled via a fake client.

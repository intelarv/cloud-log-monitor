The honorific case ("Dr. Park") was already covered by the honorific arm of `NAME_CASING_RE`. `"An"` is intentionally excluded even from the context pass — too common a word for a context anchor alone to disambiguate. New fixtures (`name-10/11/12` positives; `benign-collision-1..4` precision controls) lock the behavior; detector-phi stays re-baselined at **1.0** with zero benign false positives. The remaining **un-anchored** word-collision case ("Park reviewed the chart" with no honorific/context keyword) cannot be closed deterministically without regressing precision and is now handled by a **default-inert production NER seam** (built; off unless configured): `NER_PROVIDER=aws-comprehend|gcp-dlp|azure-language` augments ingest detection with cloud NER (AWS Comprehend / GCP DLP / Azure Language — all BAA-eligible), mirroring the embedder/search/raw-evidence factories. `lib/ner.ts` (`NerProvider` interface + `NoopNerProvider` default), `lib/ner-config.ts` (env factory/registry/`initNerProviderFromEnv`), `lib/cloud-ner.ts` (lazy per-cloud providers); `scanForPhiWithNer(text, ner)` = Stage-1 `scanForPhi` ∪ async NER spans, merged + deduped (NER spans fully covered by a Stage-1 span are dropped), wired into the async ingest path only. The default is `none` (`NoopNerProvider` returns `[]`), so **no SDK is loaded and the offline credential-free eval gate stays byte-identical** — the model dependency lives entirely behind the opt-in env, never in the gate. PHI posture is unchanged (NER reads the same trust-boundaried payload Stage-1 sees; spans masked by the same `redactInline`; only the redacted projection persisted).
  - **M13.4 (done) — Additional secret classes (prefix-/context-anchored).** `aws_secret_key` (40-char secret, anchored on `aws_secret_access_key=`/`secret_access_key=`; only the access-key ID was caught before), `stripe_key` (`[srp]k_(live|test)_…`), `twilio_sid` (`AC`/`SK`+32 hex), `sendgrid_key` (`SG.<>.<>`), `llm_api_key` (`sk-`/`sk-proj-`/`sk-ant-…`), `npm_token` (`npm_`+36), `vault_token` (`hvs.…`), `azure_storage_key` (`AccountKey=<86 base64>==`). GCP service-account JSON private keys are already covered by the existing `private_key_pem` detector (PEM body) — locked with a regression fixture rather than a new detector.
  - **M13.5 (done) — Generic high-entropy secret (last-resort).** `high_entropy_secret` (custom `scan`): a Shannon-entropy gate (`shannonEntropy` ≥3.5, value len ≥20) applied ONLY to values on secret-shaped keys (`api_key`/`secret`/`token`/`access_token`/`client_secret`/… via `HIGH_ENTROPY_KEY_RE`). Catches prefix-less base64/hex blobs the `generic_password` detector (which only covers `password`/`pwd`/`passwd`) misses; the entropy + key-context double gate keeps benign `account=`/`code=`/`serial=` values unmatched. Intentionally overlaps specific detectors (defense-in-depth) — e.g. it also fires on `token=ASIA…`.

  Production note: §11 of `docs/ARCHITECTURE.md` names a Stage-1+2 detector pipeline + dictionary/NER lookups as the production replacement for these regex detectors. M13 hardened the deterministic dev/default detector that the credential-free eval gate exercises; the NER path (the remaining un-anchored half of M13.3) remains the documented production answer for the open-class identifiers (un-anchored word-collision names, free-form addresses) that regex + gazetteer alone cannot fully cover.

## M9.3 (complete)

Three per-cloud Terraform roots under `deploy/terraform/roots/{aws,gcp,azure}/`, each consuming the M9.2 Postgres module and producing the values the M9.1 Helm overlays consume. Closes the M9 cloud-deploy series: application portability (existing) + IaC portability (this slice) = operator can stand up a HIPAA-shaped cluster on any of the three clouds from a single `tofu apply`.

Per-root contents (7 files each: `versions.tf`, `variables.tf`, `main.tf`, `postgres.tf`, `notarization.tf`, `outputs.tf`, `README.md`):

- **`aws/`** — EKS cluster (private endpoint, managed node group) + IAM OIDC provider + IRSA role for the `phi-audit-api` ServiceAccount with Bedrock `InvokeModel`/`InvokeModelWithResponseStream` (region-scoped), Secrets Manager read (scoped to the three secret ARNs), CloudWatch Logs read, and explicit `kms:Decrypt` on the notarization key (cross-account, doesn't inherit Secrets Manager's same-account default). Postgres via M9.2 module. Secrets Manager entries for `DATABASE_URL` + `SESSION_SECRET` (in-account, encrypted with `db_kms_key_id`), and `NOTARIZATION_SECRET` created via the `aws.notarization` aliased provider so the secret itself lives in the separate account alongside its KMS key; cross-account resource policy grants the production IRSA role `secretsmanager:GetSecretValue` + `DescribeSecret`. Outputs map directly to `values-aws.yaml` overlay fields (irsa_role_arn → `serviceAccount.api.annotations`, secret ARNs → ESO sync targets).
- **`gcp/`** — GKE Autopilot cluster (private nodes + private endpoint, Workload Identity pool enabled). Google Service Account `phi-audit-api` with project-level `aiplatform.user` + `cloudsql.client` + `secretmanager.secretAccessor` + `logging.viewer`, and a `roles/iam.workloadIdentityUser` binding mapping `serviceAccount:<project>.svc.id.goog[<ns>/<ksa>]` → the GSA. Postgres via M9.2 module (Cloud SQL with `auto-iam-authn`-friendly defaults; `gcp_instance_connection_name` exported for the Cloud SQL Auth Proxy sidecar args). Secret Manager entries for `DATABASE_URL` + `SESSION_SECRET` in the workload project; `NOTARIZATION_SECRET` written via the `google.notarization` aliased provider in the separate notarization project — uses `user_managed` replication so a `customer_managed_encryption.kms_key_name` can target the notarization-project Cloud KMS key per replica (auto replication doesn't support CMEK). Cross-project `secretmanager.secretAccessor` IAM member grants the api GSA read on that secret. Outputs map to `values-gcp.yaml`.
- **`azure/`** — AKS cluster with `oidc_issuer_enabled = true` + `workload_identity_enabled = true` (both required; AKS WI doesn't work with either alone). User-Assigned Managed Identity `phi-audit-api` + a single `azurerm_federated_identity_credential` binding it to `system:serviceaccount:<ns>:<ksa>` against the cluster's OIDC issuer. Postgres via M9.2 module (Flexible Server PG 16 + `azure.extensions=VECTOR`). Application Key Vault data-sourced (assumed pre-created by platform team) with role assignment `Key Vault Secrets User` for the api MI; `SESSION_SECRET` + `DATABASE_URL` written into it. `NOTARIZATION_SECRET` written via the `azurerm.notarization` aliased provider into the separate-subscription notarization vault; same `Key Vault Secrets User` role assignment, this time scoped to the notarization vault id and provider-aliased so it lands in the right subscription. Outputs map to `values-azure.yaml`.

**Separate-trust-zone enforcement.** All three roots enforce the threat_model §23.2 invariant — notarization key in a different trust zone than the workload — at the Terraform provider boundary, not at runtime config. The pattern is the same across clouds: caller passes a `providers = { <cloud>.notarization = <cloud>.notarization }` block configured to the separate account/project/subscription; the notarization-related resources (the secret itself + the cross-trust-zone role/policy grant) carry `provider = <cloud>.notarization`. Per-cloud README captures the chicken-and-egg setup (the cross-zone KMS/Key Vault grant has to be applied with the production-side principal's id in hand, so the typical flow is: apply once to get the IRSA role / GSA email / MI principal id, update the cross-zone key policy / IAM binding / role assignment, then re-apply). `kms_key_id` on the M9.2 module remains the SAME-account storage-encryption key — not collapsed with the notarization key, called out explicitly in each root's `variables.tf` description.

**Deliberately out of scope (operator-managed):**
- BYO networking (VPC + subnets + secondary ranges + delegated subnets + private DNS zones) — every root accepts subnet/VNet ids as inputs but does not provision them. This keeps the root readable; operators already have their networking module of choice.
- DNS + TLS cert provisioning at the ingress layer (cloud-specific: Route53+ACM / Google-managed certs / App Gateway TLS via Front Door).
- The notarization-side KMS key / Key Vault itself — pre-created in the separate account/project/subscription. Each README documents the exact `gcloud kms keys create` / AWS key-policy / Azure RBAC mode prereqs.
- Azure OpenAI deployment + endpoint — operator-provisioned out-of-band; `values-azure.yaml` carries the deployment name.
- CI/CD pipelines (CodePipeline / Cloud Build / Azure Pipelines).

**Validation:** `pnpm run tf:fmt` (driven by `deploy/scripts/tf-fmt-check.sh`) runs `tofu fmt -recursive -check -diff` across all of `deploy/terraform/`, including the new roots. `tofu validate` deferred to CI (needs `tofu init` + ~500MB of provider plugins). No changes to tests (TF roots are deploy-time artifacts, not runtime code); test count stays **226**.

Net: with M9.1 (Helm chart), M9.2 (Postgres module), M9.3 (per-cloud roots), M9.4 (Replit deploy), and M9.5 (chat-agent runtime seam) all in, the M9 cloud-deploy series is closed. Application portability (cloud LLM adapters, cloud embedder factory, cloud log sources, chat runtime seam) + IaC portability (per-cloud roots, all consuming one Postgres module, all producing values for one Helm chart) means an operator can stand up PHI-Audit on any of EKS/GKE/AKS from a single `tofu apply` + `helm install`.

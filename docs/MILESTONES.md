This is **static** per-point selection; the input-based **LLM router** (pick a model per request from the *content*) is explicitly deferred вЂ” see the deferred survey below.

## M18 вЂ” Chat working memory + context-window management

Closes the missing short-term-memory technique for the chat agent. Conversation turns were already persisted to `chat_messages` but never fed back to the LLM, so every turn started cold; the agent could not "remember" anything earlier in the same session. M18 replays the persisted conversation into the model under a token + turn budget, with an opt-in rolling summary to preserve context that falls out of the window. New pure `lib/chat-memory.ts`: `getChatMemoryConfigFromEnv` (`CHAT_MEMORY_TOKEN_BUDGET` default 1500 / `CHAT_MEMORY_MAX_TURNS` default 20 / `CHAT_MEMORY_SUMMARY` switch default off / `CHAT_MEMORY_SUMMARY_MAX_MESSAGES` default 40 / `CHAT_MEMORY_SUMMARY_MODEL`); `approxTokens` (в‰€chars/4); `selectWindow(messages, cfg) в‡’ {window, overflow}` (keeps newest under BOTH budgets, always keeps the most-recent message even if it alone exceeds the token budget, window stays chronological); `toHistoryTurns(window, summary?)` (`user`в†’user / `assistant`в†’model, prepends a `<CONVERSATION_SUMMARY>` context turn-pair when a summary is present); `buildChatSummaryPrompt` + `summarizeChatOverflow` (injected runtime в†’ `{text, modelId}`; no DB, no implicit LLM). Schema: `chat_sessions` gains `memory_summary` (text, null) + `memory_summary_covered_count` (int, default 0) + `memory_summary_model` (text, null) вЂ” idempotent `ADD COLUMN IF NOT EXISTS` in `setup-sql.ts` (provisioned by `db setup`, not push; `chat_sessions` already RLS-enabled). Wiring: `chat-agent.ts` adds an optional `priorHistory?: LlmHistoryTurn[]` to `RunChatTurnOpts`, seeding `runAgentLoop`'s `history` (omitted в‡’ empty в‡’ injected-runtime/test path unchanged). `routes/chat.ts` loads the session's prior messages (asc, capped at 500, role-filtered) BEFORE inserting the new user message, builds the window, and threads `priorHistory` into `runChatTurn`. The replayed text is already PHI-safe (input refused / output scanned before persistence), so the window introduces no new raw PHI. **Rolling summary (the only part that calls an LLM, gated behind `CHAT_MEMORY_SUMMARY`):** overflow not yet covered by `memory_summary_covered_count` is folded into the rolling summary via `resolveLlmForDecisionPoint("chat", вЂ¦)` (M17 seam), `scanForPhi`-re-scanned before persist вЂ” clean output is persisted + ledgers `chat.memory_summarized` (folded/covered counts + `model_id` only), while a PHI hit / empty output / runtime throw keeps the prior summary and ledgers `chat.memory_summary_failed` (reason only). Both new events are NOT_ALERTABLE. **Default-safe:** a fresh single-turn session has an empty window в‡’ byte-identical to pre-feature; `CHAT_MEMORY_SUMMARY` unset в‡’ no extra LLM call; the credential-free eval gate never exercises persisted multi-turn chat в‡’ byte-identical. Tests: new `chat-memory.test.ts` (16 cases вЂ” token math, window budget/cap/order/overflow edge cases, role-map + summary-prepend, env config defaults + switch parsing, prompt building, fake-runtime summarize) + a `chat-agent.test.ts` case proving `priorHistory` seeds the loop's history. **Deferred (out of scope):** a smarter per-turn retrieval-over-history memory (semantic recall of the most-relevant prior turns) rather than a fixed window + linear summary fold. **вЂ” closed by M19.**

## M19 вЂ” Chat semantic memory recall

Closes the M18 deferred item. M18 replayed only the most-RECENT turns (a fixed sliding window); a relevant turn earlier in a long session fell out of the window and the only fallback was the linear summary fold. M19 adds an opt-in **retrieval-over-history** mode that replays the most-RELEVANT prior turns via pgvector cosine over per-message embeddings. New pure helper in `lib/chat-memory.ts`: `assembleRecallWindow(allMessagesWithIds, relevantIds, cfg) в‡’ {window, overflow}` = (semantic top-K в€Є the last `recencyTail` messages), deduped, restored to chronological order, bounded by the SAME `CHAT_MEMORY_TOKEN_BUDGET` + `CHAT_MEMORY_MAX_TURNS` as recency mode (walks newestв†’oldest so the current question's immediate context is never dropped for an older relevant turn). Config gains `CHAT_MEMORY_SEMANTIC_RECALL` (switch, default off) / `CHAT_MEMORY_SEMANTIC_RECALL_K` (default 8, positive) / `CHAT_MEMORY_SEMANTIC_RECALL_RECENCY_TAIL` (default 2, в‰Ґ0). New `lib/chat-recall.ts` (DB + embedder glue): `embedAndStoreChatMessage` (upsert one row per message into `chat_message_embeddings`, `ON CONFLICT (message_id)`) + `semanticRecallMessageIds` (cosine `<=>` top-K scoped to tenant + session + the **current** `embedder_version`, stable secondary sort on `message_id`; returns `[]` for a non-positive k / empty query so the caller falls back). Schema: new `chat_message_embeddings` table in `setup-sql.ts` (`message_id` PK в†’ `chat_messages` `ON DELETE CASCADE`, `session_id`, `tenant_id`, `embedding vector(dim)`, `embedder_version`, `updated_at`; ivfflat cosine index + tenant/session indexes; RLS + FORCE RLS + `tenant_isolation`) вЂ” idempotent `CREATE TABLE IF NOT EXISTS`, provisioned by `db setup`, not push. Wiring in `routes/chat.ts`: when the switch is on, after persisting each user + assistant message it embeds the (already-PHI-safe) content best-effort; on each turn it embeds the current question, recalls the top-K, and `assembleRecallWindow` builds the replayed window (any failure в†’ fall back to the recency `selectWindow`). **Embedder:** the shared `getEmbedder()` (PhiGuard-wrapped) вЂ” the offline `FeatureHashEmbedder` by default, so it works credential-free with no cloud SDK. **Orthogonal to the M18 summary:** the rolling summary still folds the RECENCY overflow (monotonic `coveredCount` preserved); semantic recall only swaps the window passed to `toHistoryTurns`, so the two compose (summary = old dropped context; recall = the most-relevant specific turns). **PHI posture:** `chat_messages.content` is already PHI-safe and the embedder is PhiGuard-wrapped, so no raw PHI reaches the vector tier; **no new ledger events and no LLM call** (pure retrieval). **Default-safe:** switch off в‡’ byte-identical to M18 (no embedding writes, no recall query); a fresh single-turn session has an empty window regardless; the credential-free eval gate never runs persisted multi-turn chat в‡’ byte-identical. Tests: new `assembleRecallWindow` cases + config defaults/switch in `chat-memory.test.ts` (26 cases total) + new `chat-recall.test.ts` DB integration suite (store + idempotent upsert, similarity ranking with `FeatureHashEmbedder`, k/empty-query guard, `embedder_version` filter, tenant+session scoping). **Deferred (out of scope):** per-tenant LIST partitioning of `chat_message_embeddings` (mirrors the `finding_embeddings` single-table default); a hybrid BM25 в€Є vector fuse over chat history (recall is vector-only for v1).

## M9.3 (complete)

Three per-cloud Terraform roots under `deploy/terraform/roots/{aws,gcp,azure}/`, each consuming the M9.2 Postgres module and producing the values the M9.1 Helm overlays consume. Closes the M9 cloud-deploy series: application portability (existing) + IaC portability (this slice) = operator can stand up a HIPAA-shaped cluster on any of the three clouds from a single `tofu apply`.

Per-root contents (7 files each: `versions.tf`, `variables.tf`, `main.tf`, `postgres.tf`, `notarization.tf`, `outputs.tf`, `README.md`):

- **`aws/`** вЂ” EKS cluster (private endpoint, managed node group) + IAM OIDC provider + IRSA role for the `phi-audit-api` ServiceAccount with Bedrock `InvokeModel`/`InvokeModelWithResponseStream` (region-scoped), Secrets Manager read (scoped to the three secret ARNs), CloudWatch Logs read, and explicit `kms:Decrypt` on the notarization key (cross-account, doesn't inherit Secrets Manager's same-account default). Postgres via M9.2 module. Secrets Manager entries for `DATABASE_URL` + `SESSION_SECRET` (in-account, encrypted with `db_kms_key_id`), and `NOTARIZATION_SECRET` created via the `aws.notarization` aliased provider so the secret itself lives in the separate account alongside its KMS key; cross-account resource policy grants the production IRSA role `secretsmanager:GetSecretValue` + `DescribeSecret`. Outputs map directly to `values-aws.yaml` overlay fields (irsa_role_arn в†’ `serviceAccount.api.annotations`, secret ARNs в†’ ESO sync targets).
- **`gcp/`** вЂ” GKE Autopilot cluster (private nodes + private endpoint, Workload Identity pool enabled). Google Service Account `phi-audit-api` with project-level `aiplatform.user` + `cloudsql.client` + `secretmanager.secretAccessor` + `logging.viewer`, and a `roles/iam.workloadIdentityUser` binding mapping `serviceAccount:<project>.svc.id.goog[<ns>/<ksa>]` в†’ the GSA. Postgres via M9.2 module (Cloud SQL with `auto-iam-authn`-friendly defaults; `gcp_instance_connection_name` exported for the Cloud SQL Auth Proxy sidecar args). **Separate-trust-zone enforcement.** All three roots enforce the threat_model В§23.2 invariant вЂ” notarization key in a different trust zone than the workload вЂ” at the Terraform provider boundary, not at runtime config. - CI/CD pipelines (CodePipeline / Cloud Build / Azure Pipelines).

# Configuration reference

Full operator reference for environment variables, optional subsystems, and the
eval gate. `replit.md` keeps a one-line summary of each of these and links here
for the detail.

## Required env

- `DATABASE_URL`, `SESSION_SECRET`

## Eval gate (detail)

- `pnpm --filter @workspace/api-server run eval` â€” on-demand eval suite (separate from `test`); runs deterministic detector/agent quality suites + regression gate vs `evals/baseline.json`. `eval:update` rewrites the baseline; `EVAL_LLM=1` also runs the live LLM/DB suites. See `artifacts/api-server/evals/README.md`.
- `pnpm run eval:gate` â€” credential-free CI quality gate (`deploy/scripts/eval-gate.sh`): runs the six deterministic eval suites + `evals/gate.mjs` and fails on any >5pt regression. Registered as the `eval-gate` validation command so it runs automatically on changes. The wrapper injects placeholder `DATABASE_URL`/`SESSION_SECRET` (deterministic suites import `lib/db` but never connect) and unsets `EVAL_LLM`.
- `pnpm run eval:gate:llm` â€” nightly/manual job (`deploy/scripts/eval-gate-llm.sh`) that adds the `EVAL_LLM=1` live LLM/DB suites; needs a real `DATABASE_URL` + configured LLM runtime. Scheduled in production by the Helm `CronJob` (`evalGate.nightly`, off by default) or a Replit Scheduled Deployment â€” see `deploy/README.md` â†’ "Nightly eval gate". Hard-fail policy: deterministic regressions + live-suite execution failures always fail; live-suite *score* regressions are warnings unless `EVAL_LLM_MIN_SCORE` (an absolute floor 0..1) is set. On a hard-fail it posts a concise summary (per-suite scores + which check tripped) to any configured channel, reusing the same `CHANNEL_*` config + severity gating as finding alerts (maps to `high`; inert when unconfigured; scores/suite-names only, no PHI). `gate.mjs` writes the verdict to `evals/results/gate-summary.json`; `evals/notify.mjs` (pure ESM) reads it and sends.

## Optional env (by subsystem)

- Optional embedder config (see "Embedder selection" below): `EMBEDDING_PROVIDER`, `DEPLOYMENT_TARGET`, `EMBEDDING_MODEL`, `EMBEDDING_DIM` plus provider-specific creds.
- Optional lexical search provider (M10.1): `SEARCH_PROVIDER=postgres|opensearch` (default `postgres`). `opensearch` requires `OPENSEARCH_ENDPOINT` (+ optional `OPENSEARCH_INDEX_PREFIX` default `phi-audit-findings`, `OPENSEARCH_USERNAME`/`OPENSEARCH_PASSWORD` for basic auth). SDK (`@opensearch-project/opensearch`) is lazy-loaded; operators must `pnpm --filter @workspace/api-server add @opensearch-project/opensearch` before enabling. **No `DEPLOYMENT_TARGET` shortcut** â€” OpenSearch always needs an explicit endpoint, so a bare cloud target must not silently flip the lexical leg off Postgres. Only the BM25 (lexical) retriever leg moves; the pgvector semantic leg and RRF fusion are unchanged. Boot reconciles the external index after embedding backfill (no-op for Postgres).
- Optional external-index reindex batch size (M10.1): `SEARCH_REINDEX_BATCH_SIZE` (default `500`, clamped `1..10000`; blank / non-numeric / non-integer / out-of-range â‡’ default). Drives the keyset-paginated page size used both by the boot reconcile (`reconcileSearchIndex`) and the on-demand operator command. The full external-index (OpenSearch) rebuild is **batched + resumable**: it scans findings per tenant ordered by id, mirrors each page in one `_bulk` round-trip (falls back to per-doc `index` if the client lacks `_bulk`), logs progress per batch, and accepts a `--since-tenant=<tenantId> --since-id=<findingId>` resume token so an interrupted run picks up where it stopped. Because tenants are scanned in tenant_id order (not global id order), resume needs **both** the tenant and the last id from the previous run's final progress line â€” a bare id would be ambiguous across tenants (and is rejected). Reads go through the safe projection only â€” raw evidence never reaches the searchable tier. Operator command: `pnpm --filter @workspace/api-server run reindex:search -- [--batch-size=N] [--since-tenant=<tenantId> --since-id=<findingId>]` (requires `SEARCH_PROVIDER=opensearch` + `OPENSEARCH_*`; a no-op that exits 0 under Postgres FTS, whose generated tsv column is always in sync). **Default-inert:** affects only the OpenSearch path; dev (Postgres) boot + the offline eval gate are byte-identical.
- Optional per-tenant OpenSearch index isolation (M10.1): `OPENSEARCH_PER_TENANT_INDEX` (default **off** â€” accepts `1`/`true`/`yes`/`on`, case-insensitive; anything else/unset â‡’ off). **Off** keeps the pre-isolation behavior byte-for-byte: every tenant's findings live in one shared index (`OPENSEARCH_INDEX_PREFIX`) scoped only by the mandatory per-query `tenant_id` term filter. **On** routes each tenant to a physically separate index `${OPENSEARCH_INDEX_PREFIX}-${safeTenantSegment(tenantId)}`, where `safeTenantSegment` lowercases the tenant id and replaces every char outside `[a-z0-9_-]` with `-` (bounded to 40 chars for readability) then appends a 12-hex-char SHA-256 slice of the *raw* id â€” the hash makes the segment collision-free, so two tenant ids that sanitize to the same readable form (e.g. `a/b` vs `a-b`) can never co-mingle in one index. The `tenant_id` term filter is **still applied** as defense-in-depth even when each tenant already has its own index. Only affects the OpenSearch path; Postgres FTS is unchanged. **Operator-deferred (cluster side):** index-template/alias creation policy and per-index RBAC (so a tenant index can only be read by that tenant's service identity) are cluster-administration concerns, like service-mesh mTLS and per-tenant KMS lifecycle â€” the app addresses the right physical index and keeps the term filter; the cluster enforces access control. Default-inert: unset â‡’ shared index, dev + offline eval gate byte-identical.
- Optional OpenSearch ILM/ISM tiering (M30): `OPENSEARCH_ILM_ENABLED` (default **off** â€” accepts `1`/`true`/`yes`/`on`, case-insensitive; anything else/unset â‡’ off). **On** (and only when `SEARCH_PROVIDER=opensearch`) the app provisions an OpenSearch ISM (Index State Management) policy that auto-tiers the redacted-only mirror index through **hot â†’ warm â†’ cold â†’ [delete]** so lifecycle is in-app config instead of a cluster-operator-only runbook. Knobs (all ages are OpenSearch duration strings, e.g. `7d`/`30d`/`12h`): `OPENSEARCH_ILM_HOT_MAX_AGE` (hotâ†’warm transition age), `OPENSEARCH_ILM_WARM_MAX_AGE` (warmâ†’cold age), `OPENSEARCH_ILM_DELETE_MIN_AGE` (coldâ†’delete age). **None have a default** â€” at least one of the three must be set when `OPENSEARCH_ILM_ENABLED` is on, or boot throws; any age left unset simply omits that transition (e.g. setting only `DELETE_MIN_AGE` yields a hotâ†’delete policy; leaving `DELETE_MIN_AGE` unset means the index is retained indefinitely after its last configured state). `OPENSEARCH_ILM_POLICY_NAME` (default `${OPENSEARCH_INDEX_PREFIX}-ilm`, i.e. `phi-audit-findings-ilm` when the prefix is also unset). The policy carries an `ism_template` that auto-attaches it to new indices matching the `OPENSEARCH_INDEX_PREFIX*` pattern; the app issues `PUT _plugins/_ism/policies/<name>` once per provider instance (guarded), around `ensureIndex`. Only affects the OpenSearch path; Postgres FTS is unchanged. **Operator note:** warm/cold tiers require the cluster to actually have warm/cold nodes (or UltraWarm/cold storage on managed OpenSearch); without them the transitions are no-ops at the cluster level â€” the policy is correct but the physical tiering depends on cluster topology. **Default-inert:** unset â‡’ no ILM config parsed, no ISM request, search path byte-identical; the whole OpenSearch path is itself inert unless `SEARCH_PROVIDER=opensearch`, so dev + offline eval gate are byte-identical.
- Optional raw-evidence store (M10.2/M10.3): `RAW_EVIDENCE_PROVIDER=database|s3|gcs|azure-blob` (default `database` â€” raw stays inline in `findings.raw_evidence`). `s3` requires `RAW_EVIDENCE_S3_BUCKET` + `AWS_REGION` (+ optional `RAW_EVIDENCE_S3_PREFIX` default `raw-evidence`, `RAW_EVIDENCE_OBJECT_LOCK_MODE=COMPLIANCE|GOVERNANCE` default `COMPLIANCE`, `RAW_EVIDENCE_RETENTION_DAYS` default `2555`). `GOVERNANCE` is bypassable via `s3:BypassGovernanceRetention` so it does NOT give true WORM immutability for raw PHI â€” selecting it emits a boot WARN; keep `COMPLIANCE` in production unless it's an approved, time-bounded exception. `gcs` requires `RAW_EVIDENCE_GCS_BUCKET` (WORM via operator-locked bucket retention policy). `azure-blob` requires `RAW_EVIDENCE_AZURE_CONTAINER` + `RAW_EVIDENCE_AZURE_CONNECTION_STRING` (WORM via operator-locked container immutability policy). Cloud SDKs (`@aws-sdk/client-s3`, `@google-cloud/storage`, `@azure/storage-blob`) are lazy-loaded; operators must `pnpm --filter @workspace/api-server add <sdk>` before enabling. **No `DEPLOYMENT_TARGET` shortcut** â€” moving raw PHI is always explicit (needs a bucket/container). External stores write each occurrence as a NEW immutable object and record `{first,latest}` URIs in `findings.raw_evidence_ref` (jsonb); the `raw_evidence` column stays NULL. Break-glass read resolves the ref through the store (tenant + bucket re-validated on the URI); if the ref can't be resolved (malformed / no store configured / get fails) it falls back to the legacy inline `raw_evidence` column when present (mixed-state rows during a provider transition), surfacing `raw_unresolved` only when no fallback exists. `raw_evidence_ref` is excluded from `findingSafeColumns` + the redacted view, exactly like `raw_evidence`.
- Optional raw-evidence tiering lifecycle (ages already-stored inline raw PHI out of the hot `findings.raw_evidence` column into the external WORM store): `RAW_EVIDENCE_TIER_AGE_DAYS` (opt-in; **unset â†’ disabled**, the only switch that turns the job on) + `RAW_EVIDENCE_TIER_INTERVAL_MS` (scheduler cadence, default `3600000` = 1h) + `RAW_EVIDENCE_TIER_BATCH_SIZE` (max findings migrated per tenant per run, default `100`). A non-positive/non-numeric age throws at boot. **Default-INERT in two layers**: nothing is scheduled unless (a) `RAW_EVIDENCE_TIER_AGE_DAYS` is set AND (b) an EXTERNAL `RAW_EVIDENCE_PROVIDER` (`s3`/`gcs`/`azure-blob`) is active â€” the inline `database` store has no WORM tier to age into, so dev boot + the offline eval gate are byte-identical (a WARN is logged if the age is set but the store is still inline). **Why it exists:** `RAW_EVIDENCE_PROVIDER` only routes *new* ingests to the WORM store; findings written before an operator switches providers keep their raw PHI inline forever. This leader-locked job (single Postgres advisory lock, one pod tiers per cadence) scans each tenant for findings whose `last_seen_at` is older than the age window, writes the `{first,latest}` snapshots to the store as immutable objects, **verifies they are readable (get-after-put) before nulling the inline copy** (never destroys the sole copy of raw PHI), seats `findings.raw_evidence_ref`, and ledgers `raw_evidence.tiered` (NOT_ALERTABLE â€” routine, *reduces* exposure). Any put/get/verify failure leaves the inline copy intact, ledgers `raw_evidence.tier_failed` (warning), and retries next cadence. Break-glass reads are unaffected (they resolve `raw_evidence_ref` first, inline fallback otherwise). Ledger payloads + logs carry finding id + provider name only â€” never raw PHI, never object URIs (those live in `raw_evidence_ref`, out of `findingSafeColumns`).
- Optional external raw-evidence write retry (bounds transient WORM `store.put` blips before declaring durable capture degraded): `RAW_EVIDENCE_WRITE_MAX_ATTEMPTS` (default `3`, clamped `1..10`; `1` disables retry) + `RAW_EVIDENCE_WRITE_BACKOFF_MS` (base delay, default `100`, clamped `0..60000`). The external WORM write at ingest retries with exponential backoff (`base Ã— 2^(attempt-1)`, each computed delay capped at 60s) before falling through to the failure path that latches the provider/tenant degraded and ledgers `ingest.raw_evidence_store_failed` (so most throttling / brief-5xx blips recover automatically and don't page on-call). Recovery is paired and edge-triggered: when a subsequent WORM write for that provider/tenant succeeds after a prior failure, the latch clears and a single `ingest.raw_evidence_store_recovered` event is ledgered (registered `warning` in `alerts.ts`) so on-call sees the all-clear without polling; steady-state success after success emits nothing. **Default-INERT**: only reachable when an EXTERNAL `RAW_EVIDENCE_PROVIDER` (`s3`/`gcs`/`azure-blob`) is active â€” the inline `database` store can't fail this way â€” so dev boot + the offline eval gate are byte-identical regardless of these values. The retry runs OUTSIDE the dedup advisory lock, so backoff never serializes concurrent ingests of the same fingerprint. Misconfigured (blank / non-numeric / out-of-range) values fall back to the clamped default rather than disabling the retry.
- Optional memory consolidation + importance-decay eviction over pgvector (bounds the `finding_embeddings` derived cache): `MEMORY_MAX_EMBEDDINGS_PER_TENANT` (opt-in; **unset â†’ disabled**, the only switch that turns the job on; the per-tenant count cap) + `MEMORY_DECAY_HALF_LIFE_DAYS` (recency half-life for the importance score, default `30`) + `MEMORY_EVICT_INTERVAL_MS` (scheduler cadence, default `21600000` = 6h). A non-positive/non-numeric cap/half-life/interval throws at boot. **Default-INERT**: nothing is scheduled and `backfillEmbeddings` is byte-identical unless `MEMORY_MAX_EMBEDDINGS_PER_TENANT` is set, so dev boot + the offline eval gate are unchanged. **Why it exists:** `finding_embeddings` is a DERIVED cache (one pgvector row per finding, rebuilt from the redacted safe projection) that grows unbounded as findings accumulate; only the *vector* leg of hybrid retrieval reads it (the lexical/BM25 leg + break-glass raw reads are unaffected by eviction), so the cache can be pruned to bound memory/cost without touching the append-only findings audit record. This leader-locked job (single Postgres advisory lock, one pod evicts per cadence) prunes two ways, both deterministic (no LLM): (1) **consolidation** â€” among OLD (older than the half-life) or RESOLVED/false-positive findings sharing the same `(classification, subclass, source)`, keep only the single highest-importance representative's embedding and drop the duplicates; recent open findings are never consolidated; (2) **count cap** â€” after consolidation keep at most `MEMORY_MAX_EMBEDDINGS_PER_TENANT` embeddings per tenant (top-N by importance). **Hard floor:** a critical-severity, still-open finding is NEVER evicted by either pass, regardless of age or cap. Importance is a pure deterministic score: severity weight Ã— exponential recency decay on `last_seen_at` (the configured half-life) + a log-damped occurrence-count bonus âˆ’ a resolved/false-positive penalty. **One shared policy gate** is consulted by BOTH `backfillEmbeddings` (refrains from CREATING an embedding for a finding the policy would evict) AND the eviction job (REMOVES ineligible embeddings), so boot backfill never recreates what eviction removed (no recreate thrash). New event types: `memory.evicted` (NOT_ALERTABLE â€” routine cache maintenance) + `memory.evict_failed` (warning â€” embeddings left intact, retried next cadence). Ledger payloads + logs carry counts + policy parameters only â€” never PHI, snippets, or finding ids. **No schema change** (`finding_embeddings`, `last_seen_at`, `occurrence_count` all pre-exist).
- Optional LLM-generated cross-finding consolidation summaries (rides the same leader-locked memory job): `MEMORY_CONSOLIDATION_SUMMARY` (opt-in; **unset â†’ disabled**, the only switch that turns it on) + `MEMORY_SUMMARY_MAX_GROUPS_PER_RUN` (per-tenant LLM-call cap per sweep, default `20`) + `MEMORY_SUMMARY_MAX_MEMBERS_SAMPLED` (max group members whose redacted snippet is sampled into the prompt, default `8`) + `MEMORY_SUMMARY_MODEL` (model id hint, default = chat agent model). A non-positive/non-numeric cap throws at boot. **Default-INERT â€” and this is the one memory switch that calls an LLM**, so it is gated behind its OWN flag separate from `MEMORY_MAX_EMBEDDINGS_PER_TENANT`: unset â†’ no LLM call, nothing scheduled, dev boot + the offline credential-free eval gate stay byte-identical. **Why it exists:** the deterministic consolidation pass (above) collapses a `(classification, subclass, source)` cluster down to a single representative embedding but discards the human-readable "what was this cluster" context; this captures one natural-language summary per collapsed group into the new `memory_consolidation_summaries` table (RLS, tenant-scoped, unique `(tenant_id, group_key)`). **PHI posture:** the prompt is built from the **redacted safe projection only** (never `raw_evidence`), and the LLM's output is re-scanned by `scanForPhi` before persist â€” any hit **hard-fails the row** (no write) and ledgers `memory.summary_failed`. The upsert is keyed by a `member_signature` (sorted member-id hash): an unchanged group is **skipped** (no LLM call), so re-runs are idempotent and cost-bounded. Ledger payloads + logs carry **counts + `model_id` only** â€” never finding ids, snippets, or PHI. New event types: `memory.summarized` (NOT_ALERTABLE â€” routine) + `memory.summary_failed` (warning). The `group_key` is stored base64-encoded because the in-memory grouping key is NUL-joined and Postgres `text` rejects NUL bytes; the readable parts live in their own `classification`/`subclass`/`source` columns.
- Optional Stage-2 NER detector (M13.3 production path): `NER_PROVIDER=none|presidio|aws-comprehend|gcp-dlp|azure-language` (default `none` â€” `NoopNerProvider`, Stage-1 deterministic detection only, eval gate byte-identical, no SDK loaded). When set, ingest detection runs `scanForPhiWithNer` = Stage-1 `scanForPhi` âˆª async NER, merging the open-class identifiers regex + gazetteer cannot precision-match (un-anchored word-collision names like "Park reviewed the chart", free-form postal addresses). Per-provider:
  - `presidio` (self-hosted, no cloud account/BAA) â†’ `NER_PRESIDIO_ENDPOINT` (base URL of the operator-hosted Microsoft Presidio Analyzer, e.g. `http://presidio:3000`) [+ `NER_PRESIDIO_LANGUAGE` default `en`, `NER_PRESIDIO_SCORE_THRESHOLD` default `0.5`]. Reached over plain HTTP via global `fetch` â€” **no SDK to install**. POSTs the trust-boundaried payload to `${endpoint}/analyze` requesting only `PERSON`/`LOCATION`, maps PERSONâ†’person / LOCATIONâ†’address, and converts Presidio's codepoint offsets to JS UTF-16 indices. A non-OK analyzer response throws (loud, never a silent plaintext fallback). The analyzer runs in the deployment's own trust zone, so the payload never leaves it â€” which is why this backend needs no cloud BAA (operator owns analyzer TLS, same posture as the broker/object-store seams).
  - `local` (self-contained, no cloud account/BAA, **no SDK and no network**) â†’ optional tuning `NER_LOCAL_MIN_TOKEN_LEN` (default `3`, ignore shorter capitalized tokens) + `NER_LOCAL_CAPITALIZED_ONLY` (default `true`, only consider capitalized tokens as candidate names). A fully in-process gazetteer scan (`lib/local-ner.ts`) reusing the name dictionaries exported from `redact.ts`; recovers the un-anchored capitalized dictionary-name case Stage-1 omits without any external dependency. Lowest-footprint backend â€” runs anywhere the API server runs, ideal for air-gapped/offline deployments. Still default-INERT (the default is `none`; `local` is pure opt-in, eval gate byte-identical when unset).
  - `aws-comprehend` â†’ `NER_AWS_REGION` (or `AWS_REGION`) [+ `NER_LANGUAGE` default `en`]; `gcp-dlp` â†’ `NER_GCP_PROJECT_ID` (or `GOOGLE_CLOUD_PROJECT`); `azure-language` â†’ `NER_AZURE_ENDPOINT` + `NER_AZURE_API_KEY` [+ `NER_LANGUAGE` default `en`]. Cloud SDKs (`@aws-sdk/client-comprehend`, `@google-cloud/dlp`, `@azure/ai-text-analytics`) are lazy-loaded; operators must `pnpm --filter @workspace/api-server add <sdk>` before enabling (a clear runtime error names the missing dep otherwise). BAA-eligible endpoints: AWS Comprehend / GCP DLP / Azure Language under HIPAA.

  **No `DEPLOYMENT_TARGET` shortcut** â€” selecting a NER backend (and thus where log text is sent) is always explicit. PHI posture is unchanged: NER reads the same trust-boundaried log payload Stage-1 sees, its spans are masked by the same `redactInline` path, and only the redacted projection is persisted. The *sync* `scanForPhi` (Stage-1) is untouched, so the offline credential-free eval gate stays exactly as before.
- Optional agent/LLM harness limits (threat model Â§DoS â€” all have dev-safe defaults; tune for cost/latency): `LLM_CALL_TIMEOUT_MS` (default 30000) hard timeout draining a single LLM stream; `TOOL_CALL_TIMEOUT_MS` (default 10000) hard timeout per tool handler (a breach â†’ `{ok:false, code:"timeout"}`); `AGENT_MAX_TOOL_CALLS` (default 2) max tool calls chained per turn; `AGENT_MAX_OUTPUT_TOKENS_PER_TURN` (default 4096) per-turn approx-output-token circuit breaker; `AGENT_MAX_LLM_RETRIES` (default 1) + `AGENT_LLM_RETRY_DELAY_MS` (default 250) bounded retry before degrading; `AGENT_MAX_TOOL_RESULT_BYTES` (default 16384) clamp on a tool result fed back into the prompt. When the LLM is unavailable, the cost cap trips, or the tool budget is exhausted, the chat turn **degrades to a deterministic, redacted finding summary** (never an error page, never raw `tool_call` JSON); the `chat.agent_turn` ledger entry records `degraded`/`degrade_reason`/`approx_output_tokens`.
- Optional chat working memory + context-window management (gives the chat agent short-term memory by replaying the persisted conversation into the LLM): `CHAT_MEMORY_TOKEN_BUDGET` (approx-token budget for the replayed window, default `1500`) + `CHAT_MEMORY_MAX_TURNS` (max prior messages replayed, default `20`) + `CHAT_MEMORY_SUMMARY` (opt-in switch for the rolling summary â€” accepts `1`/`true`/`yes`/`on`, case-insensitive; **default off**) + `CHAT_MEMORY_SUMMARY_MAX_MESSAGES` (max overflow messages folded into the summary per turn, default `40`) + `CHAT_MEMORY_SUMMARY_MODEL` (model id hint for the summary call, default = chat agent model). A non-positive/non-numeric budget/turns/max-messages throws at boot. **What it does:** on each chat turn the route loads the session's persisted messages (already PHI-safe â€” input refused / output scanned before persistence), selects a chronological window under BOTH the token budget and the turn cap (newest kept, older overflow dropped), maps it to LLM history (`user`â†’user, `assistant`â†’model) and threads it in front of the current question via the new `priorHistory` seam on `runChatTurn`. **Rolling summary (the one part that calls an LLM, gated behind `CHAT_MEMORY_SUMMARY`):** when on, overflow not yet represented by the stored summary is folded into a rolling per-session summary (persisted on `chat_sessions.memory_summary` + `memory_summary_covered_count` + `memory_summary_model`), prepended to the window as a summary context turn-pair so older context survives window eviction. **PHI posture:** the summary prompt is built from the redacted chat text only and the LLM output is re-scanned by `scanForPhi` before persist â€” any hit (or empty output) keeps the prior summary, writes nothing, and ledgers `chat.memory_summary_failed` (reason only); a clean summary ledgers `chat.memory_summarized` (counts + `model_id` only â€” never message/summary text). Both events are NOT_ALERTABLE. **Default-INERT for the new behavior that matters:** a fresh single-turn session has no prior messages, so the window is empty and the turn is byte-identical to pre-feature; with `CHAT_MEMORY_SUMMARY` unset there is **no extra LLM call**. The credential-free eval gate never exercises persisted multi-turn chat, so it stays byte-identical. **Static window only** â€” the rolling summary is a fixed fold of overflow; a smarter per-turn retrieval-over-history memory is out of scope here.
- Optional chat **semantic memory recall** (M19) â€” replays the most-RELEVANT prior turns, not just the most-recent ones: `CHAT_MEMORY_SEMANTIC_RECALL` (opt-in switch â€” accepts `1`/`true`/`yes`/`on`, case-insensitive; **default off**) + `CHAT_MEMORY_SEMANTIC_RECALL_K` (how many semantically-closest prior messages to retrieve, default `8`, must be positive) + `CHAT_MEMORY_SEMANTIC_RECALL_RECENCY_TAIL` (how many most-recent messages to always keep regardless of similarity, default `2`, must be â‰¥0). **What it does:** when on, every persisted chat message is embedded (best-effort) and upserted into `chat_message_embeddings` (one row per message, `message_id` PK â†’ `chat_messages` `ON DELETE CASCADE`, pgvector ivfflat cosine index, RLS + FORCE RLS + tenant_isolation, dim = `EMBEDDING_DIM`); on the next turn the route embeds the current question, cosine-ranks the session's prior messages (scoped to tenant + session + the **current** `embedder_version`), and assembles the replayed window as `assembleRecallWindow` = (semantic top-K âˆª the recency tail), deduped, restored to chronological order, and bounded by the SAME `CHAT_MEMORY_TOKEN_BUDGET` + `CHAT_MEMORY_MAX_TURNS` as recency mode. **Embedder:** uses the shared `getEmbedder()` (PhiGuard-wrapped) â€” the offline `FeatureHashEmbedder` by default, so it works credential-free; no cloud SDK required. **Orthogonal to the M18 rolling summary:** the summary still folds the **recency** overflow (unchanged, monotonic `coveredCount` preserved); semantic recall only swaps the window passed to `toHistoryTurns`. **PHI posture:** `chat_messages.content` is already PHI-safe (input refused / output scanned before persistence) and the embedder is PhiGuard-wrapped, so no raw PHI reaches the vector tier. **No new ledger events / no LLM call** (pure retrieval). **Best-effort:** any embed/store/recall failure is caught and the turn falls back to the recency window. **Default-INERT:** switch off â‡’ byte-identical to M18 (no embedding writes, no recall query); The per-tenant daily LLM cost breaker (`AGENT_DAILY_TOKEN_BUDGET`, default 1,000,000 tokens/tenant/UTC-day; threat_model Â§DoS) lives behind an `AgentBudgetStore` seam. **`memory` (default):** the original per-PROCESS in-memory `Map` â€” byte-identical to the pre-seam breaker and the only store the offline eval gate exercises. Its limitation: a multi-process deployment (e.g. `WORKFLOW_ENGINE=temporal` with N workers) gets N independent budget maps, so the effective fleet-wide cap is up to NÃ— the configured per-tenant budget. **`postgres` (opt-in):** a shared `tenant_budgets` row per `(tenant_id, UTC day)` charged via an atomic `INSERT â€¦ ON CONFLICT (tenant_id, day_key) DO UPDATE SET tokens_used = tenant_budgets.tokens_used + EXCLUDED.tokens_used` under the tenant GUC (RLS-isolated), so the per-tenant daily cap is enforced **once across the whole fleet** â€” no read-modify-write race. The table holds counters only (tenant id, calendar day, integer total) â€” no PHI. Default-INERT: unset/`memory` â‡’ eval gate byte-identical.
- Optional ingest dead-letter queue (M8.1): `INGEST_DEAD_LETTER_ENABLED=1|true` (switch â€” anything else â‡’ disabled) [+ `INGEST_DLQ_MAX_ATTEMPTS` (default 3, clamped 1..10), `INGEST_DLQ_BACKOFF_MS` (base delay, default 100, clamped 0..60000; doubles each retry, each computed delay capped at 60s)]. **DLQ off (default):** the pipeline processes each record once and RETHROWS on failure so a real broker no-acks + redelivers (at-least-once) and `/api/admin/ingest/replay` surfaces the error â€” byte-identical to pre-M8.1, the catch being a genuine poison record redelivers forever. **DLQ on:** the handler retries the record up to `INGEST_DLQ_MAX_ATTEMPTS` with exponential backoff, and on terminal failure persists a **metadata-only** marker to `ingest_dead_letter` (source pointer + `payload_sha256` + byte count + error *name* + attempt count â€” never the raw, possibly-PHI payload), ledgers `ingest.dead_lettered` (high), and ACKs so the broker stops redelivering the poison. The dead-letter write is best-effort â€” if it fails, the handler falls through to the original rethrow so nothing is silently lost. Default-INERT: unset â‡’ eval gate byte-identical.

## Embedder selection (cloud-aware)

The embedder used for hybrid retrieval is selected at boot via env. Precedence:
`EMBEDDING_PROVIDER` > `DEPLOYMENT_TARGET` > default (`featurehash`).

| Provider | Default model | Native HIPAA story | Required env |
|---|---|---|---|
| `featurehash` (dev) | â€” | n/a (local) | none |
| `bedrock` (AWS) | `amazon.titan-embed-text-v2:0` | BAA via AWS | `AWS_REGION` + SDK credential chain |
| `vertex` (GCP) | `text-embedding-005` | BAA via GCP | `GCP_PROJECT_ID`, `GCP_LOCATION`, ADC |
| `azure-openai` (Azure) | `text-embedding-3-small` | BAA via Azure | `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_DEPLOYMENT` |
| `tei` (cloud-agnostic) | server-side (e.g. `nomic-embed-text-v1.5`, `Qwen3-Embedding-0.6B`, `embeddinggemma-300m`) | data never leaves cluster | `TEI_ENDPOINT` |

`DEPLOYMENT_TARGET=aws|gcp|azure|local` is a shortcut that picks the cloud-native provider above; explicit `EMBEDDING_PROVIDER` always wins.

`EMBEDDING_DIM` (default `256`) sets both the embedder output dim and the `finding_embeddings.embedding` column dim. All four cloud providers' defaults support Matryoshka truncation to 256 natively. Changing dim on an existing DB: `DROP TABLE finding_embeddings; restart` â€” embeddings are a cache, backfill rebuilds them. The boot path validates that the existing column dim matches the configured one and fails loudly otherwise.

### Per-tenant pgvector partitioning (M12.2)

`EMBEDDINGS_TENANT_PARTITIONING` (default off; accepts `1|true|on|yes` / `0|false|off|no`/empty) selects the `finding_embeddings` storage layout at boot:

- **off (default)** â€” a single `finding_embeddings` table keyed by `finding_id`. Byte-identical to pre-M12.2; this is the dev + credential-free eval-gate path.
- **on** â€” `finding_embeddings` is `PARTITION BY LIST (tenant_id)` with a composite PK `(finding_id, tenant_id)`, an ivfflat index per partition, and a catch-all `finding_embeddings_default DEFAULT` partition so any un-provisioned tenant still stores + searches correctly. The conversion is idempotent: boot only drops + recreates the table when it exists and is not yet partitioned, then the embedding backfill rebuilds the rows (embeddings are a derived cache, so this is non-destructive).

Opt-in dedicated per-tenant partitions are provisioned with `provisionTenantEmbeddingPartition(tenantId)` (`lib/db/tenant-partition.ts`): it creates `finding_embeddings_<md5(tenantId)>` for an explicit `tenant_id`, is a no-op unless partitioning is on, and validates `tenant_id` against `SAFE_TENANT_ID`.

### Per-tenant chat-embeddings partitioning (M22)

`CHAT_EMBEDDINGS_TENANT_PARTITIONING` (default off; same `1|true|on|yes` / `0|false|off|no`/empty parsing) mirrors the M12.2 switch above for the M19 `chat_message_embeddings` table, and is **fully independent** of `EMBEDDINGS_TENANT_PARTITIONING` (its own env var + its own module state â€” flipping one does not flip the other). It selects the `chat_message_embeddings` storage layout at boot:

- **off (default)** â€” a single `chat_message_embeddings` table keyed by `message_id`. Byte-identical to M19; this is the dev + credential-free eval-gate path.
- **on** â€” `chat_message_embeddings` is `PARTITION BY LIST (tenant_id)` with a composite PK `(message_id, tenant_id)`, an ivfflat cosine index per partition, and a catch-all `chat_message_embeddings_default DEFAULT` partition so any un-provisioned tenant still stores + recalls correctly. The conversion is idempotent: boot only drops + recreates the table when it exists and is not yet partitioned (embeddings are a derived cache, re-seeded on demand by the recall path, so this is non-destructive).

Opt-in dedicated per-tenant partitions are provisioned with `provisionTenantChatEmbeddingPartition(tenantId)` (`lib/db/tenant-partition.ts`): it creates `chat_message_embeddings_t_<md5(tenantId)>` for an explicit `tenant_id`, is a no-op unless partitioning is on, and validates `tenant_id` against `SAFE_TENANT_ID`. The API-server switch (`lib/embeddings-partition-config.ts`) also drives the `chat-recall.ts` upsert conflict target (`(message_id)` vs `(message_id, tenant_id)`), and the boot reconcile (`reconcileChatEmbeddingsPartitioningFromDb`) trusts the live catalog over env intent.

### Per-tenant KMS keys (M12.1)

There is **no env switch** â€” per-tenant signing keys are data-driven via the `tenant_kms_keys` control-plane table (`tenant_id` â†’ `key_id` + `provider` + `key_ref`), loaded into an in-memory registry at boot. With an empty table (dev + eval-gate default) every resolve returns null and cookies sign/verify under the global `SESSION_SECRET`, byte-identical to pre-M12.1. Providers:

- `derived` â€” HKDF of the global `SESSION_SECRET` with the tenant as salt and `key_id:purpose` in info (distinct, non-colliding key per tenant/purpose; bump `key_id` to rotate). No extra dependency.
- `external` â€” `key_ref` names an **ENV VAR** holding the key material (operator wires Vault / cloud KMS â†’ env). The table stores only the variable name, never the secret, so no key material lives at rest in the DB.

Actual cloud-KMS provisioning + rotation lifecycle is operator/Terraform territory; the application side is the table, the `resolveTenantSecret(tenantId, purpose)` resolver, and the auth-path wiring.

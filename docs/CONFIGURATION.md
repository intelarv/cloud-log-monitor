# Configuration reference

Full operator reference for environment variables, optional subsystems, and the
eval gate. `replit.md` keeps a one-line summary of each of these and links here
for the detail.

## Required env

- `DATABASE_URL`, `SESSION_SECRET`

## Eval gate (detail)

- `pnpm --filter @workspace/api-server run eval` — on-demand eval suite (separate from `test`); runs deterministic detector/agent quality suites + regression gate vs `evals/baseline.json`. `eval:update` rewrites the baseline; `EVAL_LLM=1` also runs the live LLM/DB suites. See `artifacts/api-server/evals/README.md`.
- `pnpm run eval:gate` — credential-free CI quality gate (`deploy/scripts/eval-gate.sh`): runs the six deterministic eval suites + `evals/gate.mjs` and fails on any >5pt regression. Registered as the `eval-gate` validation command so it runs automatically on changes. The wrapper injects placeholder `DATABASE_URL`/`SESSION_SECRET` (deterministic suites import `lib/db` but never connect) and unsets `EVAL_LLM`.
- `pnpm run eval:gate:llm` — nightly/manual job (`deploy/scripts/eval-gate-llm.sh`) that adds the `EVAL_LLM=1` live LLM/DB suites; needs a real `DATABASE_URL` + configured LLM runtime. Scheduled in production by the Helm `CronJob` (`evalGate.nightly`, off by default) or a Replit Scheduled Deployment — see `deploy/README.md` → "Nightly eval gate". Hard-fail policy: deterministic regressions + live-suite execution failures always fail; live-suite *score* regressions are warnings unless `EVAL_LLM_MIN_SCORE` (an absolute floor 0..1) is set. On a hard-fail it posts a concise summary (per-suite scores + which check tripped) to any configured channel, reusing the same `CHANNEL_*` config + severity gating as finding alerts (maps to `high`; inert when unconfigured; scores/suite-names only, no PHI). `gate.mjs` writes the verdict to `evals/results/gate-summary.json`; `evals/notify.mjs` (pure ESM) reads it and sends.

## Optional env (by subsystem)

- Optional embedder config (see "Embedder selection" below): `EMBEDDING_PROVIDER`, `DEPLOYMENT_TARGET`, `EMBEDDING_MODEL`, `EMBEDDING_DIM` plus provider-specific creds.
- Optional lexical search provider (M10.1): `SEARCH_PROVIDER=postgres|opensearch` (default `postgres`). `opensearch` requires `OPENSEARCH_ENDPOINT` (+ optional `OPENSEARCH_INDEX_PREFIX` default `phi-audit-findings`, `OPENSEARCH_USERNAME`/`OPENSEARCH_PASSWORD` for basic auth). SDK (`@opensearch-project/opensearch`) is lazy-loaded; operators must `pnpm --filter @workspace/api-server add @opensearch-project/opensearch` before enabling. **No `DEPLOYMENT_TARGET` shortcut** — OpenSearch always needs an explicit endpoint, so a bare cloud target must not silently flip the lexical leg off Postgres. Only the BM25 (lexical) retriever leg moves; the pgvector semantic leg and RRF fusion are unchanged. Boot reconciles the external index after embedding backfill (no-op for Postgres).
- Optional external-index reindex batch size (M10.1): `SEARCH_REINDEX_BATCH_SIZE` (default `500`, clamped `1..10000`; blank / non-numeric / non-integer / out-of-range ⇒ default). Drives the keyset-paginated page size used both by the boot reconcile (`reconcileSearchIndex`) and the on-demand operator command. The full external-index (OpenSearch) rebuild is **batched + resumable**: it scans findings per tenant ordered by id, mirrors each page in one `_bulk` round-trip (falls back to per-doc `index` if the client lacks `_bulk`), logs progress per batch, and accepts a `--since-tenant=<tenantId> --since-id=<findingId>` resume token so an interrupted run picks up where it stopped. Because tenants are scanned in tenant_id order (not global id order), resume needs **both** the tenant and the last id from the previous run's final progress line — a bare id would be ambiguous across tenants (and is rejected). Reads go through the safe projection only — raw evidence never reaches the searchable tier. Operator command: `pnpm --filter @workspace/api-server run reindex:search -- [--batch-size=N] [--since-tenant=<tenantId> --since-id=<findingId>]` (requires `SEARCH_PROVIDER=opensearch` + `OPENSEARCH_*`; a no-op that exits 0 under Postgres FTS, whose generated tsv column is always in sync). **Default-inert:** affects only the OpenSearch path; dev (Postgres) boot + the offline eval gate are byte-identical.
- Optional per-tenant OpenSearch index isolation (M10.1): `OPENSEARCH_PER_TENANT_INDEX` (default **off** — accepts `1`/`true`/`yes`/`on`, case-insensitive; anything else/unset ⇒ off). **Off** keeps the pre-isolation behavior byte-for-byte: every tenant's findings live in one shared index (`OPENSEARCH_INDEX_PREFIX`) scoped only by the mandatory per-query `tenant_id` term filter. **On** routes each tenant to a physically separate index `${OPENSEARCH_INDEX_PREFIX}-${safeTenantSegment(tenantId)}`, where `safeTenantSegment` lowercases the tenant id and replaces every char outside `[a-z0-9_-]` with `-` (bounded to 40 chars for readability) then appends a 12-hex-char SHA-256 slice of the *raw* id — the hash makes the segment collision-free, so two tenant ids that sanitize to the same readable form (e.g. `a/b` vs `a-b`) can never co-mingle in one index. The `tenant_id` term filter is **still applied** as defense-in-depth even when each tenant already has its own index. Only affects the OpenSearch path; Postgres FTS is unchanged. **Operator-deferred (cluster side):** index-template/alias creation policy and per-index RBAC (so a tenant index can only be read by that tenant's service identity) are cluster-administration concerns, like service-mesh mTLS and per-tenant KMS lifecycle — the app addresses the right physical index and keeps the term filter; the cluster enforces access control. Default-inert: unset ⇒ shared index, dev + offline eval gate byte-identical.
- Optional raw-evidence store (M10.2/M10.3): `RAW_EVIDENCE_PROVIDER=database|s3|gcs|azure-blob` (default `database` — raw stays inline in `findings.raw_evidence`). `s3` requires `RAW_EVIDENCE_S3_BUCKET` + `AWS_REGION` (+ optional `RAW_EVIDENCE_S3_PREFIX` default `raw-evidence`, `RAW_EVIDENCE_OBJECT_LOCK_MODE=COMPLIANCE|GOVERNANCE` default `COMPLIANCE`, `RAW_EVIDENCE_RETENTION_DAYS` default `2555`). `GOVERNANCE` is bypassable via `s3:BypassGovernanceRetention` so it does NOT give true WORM immutability for raw PHI — selecting it emits a boot WARN; keep `COMPLIANCE` in production unless it's an approved, time-bounded exception. `gcs` requires `RAW_EVIDENCE_GCS_BUCKET` (WORM via operator-locked bucket retention policy). `azure-blob` requires `RAW_EVIDENCE_AZURE_CONTAINER` + `RAW_EVIDENCE_AZURE_CONNECTION_STRING` (WORM via operator-locked container immutability policy). Cloud SDKs (`@aws-sdk/client-s3`, `@google-cloud/storage`, `@azure/storage-blob`) are lazy-loaded; operators must `pnpm --filter @workspace/api-server add <sdk>` before enabling. **No `DEPLOYMENT_TARGET` shortcut** — moving raw PHI is always explicit (needs a bucket/container). External stores write each occurrence as a NEW immutable object and record `{first,latest}` URIs in `findings.raw_evidence_ref` (jsonb); the `raw_evidence` column stays NULL. Break-glass read resolves the ref through the store (tenant + bucket re-validated on the URI); if the ref can't be resolved (malformed / no store configured / get fails) it falls back to the legacy inline `raw_evidence` column when present (mixed-state rows during a provider transition), surfacing `raw_unresolved` only when no fallback exists. `raw_evidence_ref` is excluded from `findingSafeColumns` + the redacted view, exactly like `raw_evidence`.
- Optional raw-evidence tiering lifecycle (ages already-stored inline raw PHI out of the hot `findings.raw_evidence` column into the external WORM store): `RAW_EVIDENCE_TIER_AGE_DAYS` (opt-in; **unset → disabled**, the only switch that turns the job on) + `RAW_EVIDENCE_TIER_INTERVAL_MS` (scheduler cadence, default `3600000` = 1h) + `RAW_EVIDENCE_TIER_BATCH_SIZE` (max findings migrated per tenant per run, default `100`). A non-positive/non-numeric age throws at boot. **Default-INERT in two layers**: nothing is scheduled unless (a) `RAW_EVIDENCE_TIER_AGE_DAYS` is set AND (b) an EXTERNAL `RAW_EVIDENCE_PROVIDER` (`s3`/`gcs`/`azure-blob`) is active — the inline `database` store has no WORM tier to age into, so dev boot + the offline eval gate are byte-identical (a WARN is logged if the age is set but the store is still inline). **Why it exists:** `RAW_EVIDENCE_PROVIDER` only routes *new* ingests to the WORM store; findings written before an operator switches providers keep their raw PHI inline forever. This leader-locked job (single Postgres advisory lock, one pod tiers per cadence) scans each tenant for findings whose `last_seen_at` is older than the age window, writes the `{first,latest}` snapshots to the store as immutable objects, **verifies they are readable (get-after-put) before nulling the inline copy** (never destroys the sole copy of raw PHI), seats `findings.raw_evidence_ref`, and ledgers `raw_evidence.tiered` (NOT_ALERTABLE — routine, *reduces* exposure). Any put/get/verify failure leaves the inline copy intact, ledgers `raw_evidence.tier_failed` (warning), and retries next cadence. Break-glass reads are unaffected (they resolve `raw_evidence_ref` first, inline fallback otherwise). Ledger payloads + logs carry finding id + provider name only — never raw PHI, never object URIs (those live in `raw_evidence_ref`, out of `findingSafeColumns`).
- Optional external raw-evidence write retry (bounds transient WORM `store.put` blips before declaring durable capture degraded): `RAW_EVIDENCE_WRITE_MAX_ATTEMPTS` (default `3`, clamped `1..10`; `1` disables retry) + `RAW_EVIDENCE_WRITE_BACKOFF_MS` (base delay, default `100`, clamped `0..60000`). The external WORM write at ingest retries with exponential backoff (`base × 2^(attempt-1)`, each computed delay capped at 60s) before falling through to the failure path that latches the provider/tenant degraded and ledgers `ingest.raw_evidence_store_failed` (so most throttling / brief-5xx blips recover automatically and don't page on-call). Recovery is paired and edge-triggered: when a subsequent WORM write for that provider/tenant succeeds after a prior failure, the latch clears and a single `ingest.raw_evidence_store_recovered` event is ledgered (registered `warning` in `alerts.ts`) so on-call sees the all-clear without polling; steady-state success after success emits nothing. **Default-INERT**: only reachable when an EXTERNAL `RAW_EVIDENCE_PROVIDER` (`s3`/`gcs`/`azure-blob`) is active — the inline `database` store can't fail this way — so dev boot + the offline eval gate are byte-identical regardless of these values. The retry runs OUTSIDE the dedup advisory lock, so backoff never serializes concurrent ingests of the same fingerprint. Misconfigured (blank / non-numeric / out-of-range) values fall back to the clamped default rather than disabling the retry.
- Optional memory consolidation + importance-decay eviction over pgvector (bounds the `finding_embeddings` derived cache): `MEMORY_MAX_EMBEDDINGS_PER_TENANT` (opt-in; **unset → disabled**, the only switch that turns the job on; the per-tenant count cap) + `MEMORY_DECAY_HALF_LIFE_DAYS` (recency half-life for the importance score, default `30`) + `MEMORY_EVICT_INTERVAL_MS` (scheduler cadence, default `21600000` = 6h). A non-positive/non-numeric cap/half-life/interval throws at boot. **Default-INERT**: nothing is scheduled and `backfillEmbeddings` is byte-identical unless `MEMORY_MAX_EMBEDDINGS_PER_TENANT` is set, so dev boot + the offline eval gate are unchanged. **Why it exists:** `finding_embeddings` is a DERIVED cache (one pgvector row per finding, rebuilt from the redacted safe projection) that grows unbounded as findings accumulate; only the *vector* leg of hybrid retrieval reads it (the lexical/BM25 leg + break-glass raw reads are unaffected by eviction), so the cache can be pruned to bound memory/cost without touching the append-only findings audit record. This leader-locked job (single Postgres advisory lock, one pod evicts per cadence) prunes two ways, both deterministic (no LLM): (1) **consolidation** — among OLD (older than the half-life) or RESOLVED/false-positive findings sharing the same `(classification, subclass, source)`, keep only the single highest-importance representative's embedding and drop the duplicates; recent open findings are never consolidated; (2) **count cap** — after consolidation keep at most `MEMORY_MAX_EMBEDDINGS_PER_TENANT` embeddings per tenant (top-N by importance). **Hard floor:** a critical-severity, still-open finding is NEVER evicted by either pass, regardless of age or cap. Importance is a pure deterministic score: severity weight × exponential recency decay on `last_seen_at` (the configured half-life) + a log-damped occurrence-count bonus − a resolved/false-positive penalty. **One shared policy gate** is consulted by BOTH `backfillEmbeddings` (refrains from CREATING an embedding for a finding the policy would evict) AND the eviction job (REMOVES ineligible embeddings), so boot backfill never recreates what eviction removed (no recreate thrash). New event types: `memory.evicted` (NOT_ALERTABLE — routine cache maintenance) + `memory.evict_failed` (warning — embeddings left intact, retried next cadence). Ledger payloads + logs carry counts + policy parameters only — never PHI, snippets, or finding ids. **No schema change** (`finding_embeddings`, `last_seen_at`, `occurrence_count` all pre-exist).
- Optional LLM-generated cross-finding consolidation summaries (rides the same leader-locked memory job): `MEMORY_CONSOLIDATION_SUMMARY` (opt-in; **unset → disabled**, the only switch that turns it on) + `MEMORY_SUMMARY_MAX_GROUPS_PER_RUN` (per-tenant LLM-call cap per sweep, default `20`) + `MEMORY_SUMMARY_MAX_MEMBERS_SAMPLED` (max group members whose redacted snippet is sampled into the prompt, default `8`) + `MEMORY_SUMMARY_MODEL` (model id hint, default = chat agent model). A non-positive/non-numeric cap throws at boot. **Default-INERT — and this is the one memory switch that calls an LLM**, so it is gated behind its OWN flag separate from `MEMORY_MAX_EMBEDDINGS_PER_TENANT`: unset → no LLM call, nothing scheduled, dev boot + the offline credential-free eval gate stay byte-identical. **Why it exists:** the deterministic consolidation pass (above) collapses a `(classification, subclass, source)` cluster down to a single representative embedding but discards the human-readable "what was this cluster" context; this captures one natural-language summary per collapsed group into the new `memory_consolidation_summaries` table (RLS, tenant-scoped, unique `(tenant_id, group_key)`). **PHI posture:** the prompt is built from the **redacted safe projection only** (never `raw_evidence`), and the LLM's output is re-scanned by `scanForPhi` before persist — any hit **hard-fails the row** (no write) and ledgers `memory.summary_failed`. The upsert is keyed by a `member_signature` (sorted member-id hash): an unchanged group is **skipped** (no LLM call), so re-runs are idempotent and cost-bounded. Ledger payloads + logs carry **counts + `model_id` only** — never finding ids, snippets, or PHI. New event types: `memory.summarized` (NOT_ALERTABLE — routine) + `memory.summary_failed` (warning). The `group_key` is stored base64-encoded because the in-memory grouping key is NUL-joined and Postgres `text` rejects NUL bytes; the readable parts live in their own `classification`/`subclass`/`source` columns.
- Optional Stage-2 NER detector (M13.3 production path): `NER_PROVIDER=none|aws-comprehend|gcp-dlp|azure-language` (default `none` — `NoopNerProvider`, Stage-1 deterministic detection only, eval gate byte-identical, no SDK loaded). When set, ingest detection runs `scanForPhiWithNer` = Stage-1 `scanForPhi` ∪ async cloud NER, merging the open-class identifiers regex + gazetteer cannot precision-match (un-anchored word-collision names like "Park reviewed the chart", free-form postal addresses). Per-provider: `aws-comprehend` → `NER_AWS_REGION` (or `AWS_REGION`) [+ `NER_LANGUAGE` default `en`]; `gcp-dlp` → `NER_GCP_PROJECT_ID` (or `GOOGLE_CLOUD_PROJECT`); `azure-language` → `NER_AZURE_ENDPOINT` + `NER_AZURE_API_KEY` [+ `NER_LANGUAGE` default `en`]. Cloud SDKs (`@aws-sdk/client-comprehend`, `@google-cloud/dlp`, `@azure/ai-text-analytics`) are lazy-loaded; operators must `pnpm --filter @workspace/api-server add <sdk>` before enabling (a clear runtime error names the missing dep otherwise). **No `DEPLOYMENT_TARGET` shortcut** — sending log text to a cloud NER endpoint is always explicit (BAA-eligible endpoints: AWS Comprehend / GCP DLP / Azure Language under HIPAA). PHI posture is unchanged: NER reads the same trust-boundaried log payload Stage-1 sees, its spans are masked by the same `redactInline` path, and only the redacted projection is persisted. The *sync* `scanForPhi` (Stage-1) is untouched, so the offline credential-free eval gate stays exactly as before.
- Optional agent/LLM harness limits (threat model §DoS — all have dev-safe defaults; tune for cost/latency): `LLM_CALL_TIMEOUT_MS` (default 30000) hard timeout draining a single LLM stream; `TOOL_CALL_TIMEOUT_MS` (default 10000) hard timeout per tool handler (a breach → `{ok:false, code:"timeout"}`); `AGENT_MAX_TOOL_CALLS` (default 2) max tool calls chained per turn; `AGENT_MAX_OUTPUT_TOKENS_PER_TURN` (default 4096) per-turn approx-output-token circuit breaker; `AGENT_MAX_LLM_RETRIES` (default 1) + `AGENT_LLM_RETRY_DELAY_MS` (default 250) bounded retry before degrading; `AGENT_MAX_TOOL_RESULT_BYTES` (default 16384) clamp on a tool result fed back into the prompt. When the LLM is unavailable, the cost cap trips, or the tool budget is exhausted, the chat turn **degrades to a deterministic, redacted finding summary** (never an error page, never raw `tool_call` JSON); the `chat.agent_turn` ledger entry records `degraded`/`degrade_reason`/`approx_output_tokens`.
- Optional step-up auth (dev): `STEP_UP_DEV_TOKEN` (default `dev-stepup`, ≥8 chars). Replace with a TOTP/WebAuthn verifier in production.
- Optional notarization: `NOTARIZATION_SECRET` (≥16 chars, held in a *separate* trust zone from `SESSION_SECRET` — KMS in a separate cloud account per §23.2). Unset → SESSION_SECRET-derived dev fallback + boot WARN. Rotation: set `NOTARIZATION_RETIRED_KEYS={"<old_key_id>":"<old_secret>"}` so existing checkpoints keep verifying after `ACTIVE_KEY_ID` is bumped.
- Optional channel adapters (M6): `CHANNEL_SLACK_WEBHOOK_URL` [+ `CHANNEL_SLACK_MIN_SEVERITY=warning|high|critical`]; `CHANNEL_WEBHOOK_URL` + `CHANNEL_WEBHOOK_SECRET` (≥16) + `CHANNEL_WEBHOOK_ALLOWED_HOSTS` (CSV) [+ `CHANNEL_WEBHOOK_MIN_SEVERITY`]; `CHANNEL_PAGERDUTY_ROUTING_KEY` (Events API v2 integration key) [+ `CHANNEL_PAGERDUTY_MIN_SEVERITY`, `CHANNEL_PAGERDUTY_EVENTS_URL` (EU/custom endpoint), `CHANNEL_PAGERDUTY_DEDUP_KEY`]; `CHANNEL_RATE_LIMIT_PER_MINUTE` (default 30). Channels are inert without these. Severity maps onto PagerDuty's enum (warning→warning, high→error, critical→critical).
- Optional cloud LLM runtime: `LLM_PROVIDER=bedrock|vertex|azure-openai` (or `DEPLOYMENT_TARGET=aws|gcp|azure` shortcut). Per-provider: `bedrock` → AWS chain + `AWS_REGION` (+ `LLM_DEFAULT_MODEL`, default Claude 3.5 Haiku); `vertex` → `GCP_PROJECT_ID` + `GCP_LOCATION` (default `us-central1`) + ADC; `azure-openai` → `AZURE_OPENAI_ENDPOINT` + `AZURE_OPENAI_API_KEY` + `AZURE_OPENAI_DEPLOYMENT` (+ `AZURE_OPENAI_API_VERSION`). SDKs lazy-loaded (`@aws-sdk/client-bedrock-runtime`, `google-auth-library`); Azure is pure fetch.
- Optional **per-decision-point LLM selection (M17)**: run each of the four LLM "decision points" — `chat` (chat agent), `triage` + `verifier` (specialist agents), `summary` (memory consolidation summarizer) — on a **fully different provider + model** (e.g. `verifier` → Bedrock Claude, `chat` → Gemini). Each point reads a **scoped env overlay** `LLM_<POINT>_<KEY>` (POINT ∈ `CHAT|TRIAGE|VERIFIER|SUMMARY`) that overrides the matching global key for that point only; anything not overridden falls back to the global value. Overlayable keys (suffixes): `PROVIDER` (→ `LLM_PROVIDER`), `MODEL` (→ `LLM_DEFAULT_MODEL`), `AWS_REGION`, `GCP_PROJECT_ID`, `GCP_LOCATION`, `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_DEPLOYMENT`, `AZURE_OPENAI_API_VERSION`. Example: `LLM_VERIFIER_PROVIDER=bedrock` + `LLM_VERIFIER_AWS_REGION=us-west-2` (+ optional `LLM_VERIFIER_MODEL=anthropic.claude-3-5-sonnet-20241022-v2:0`) routes only the Verifier to Bedrock while the rest stay on the global runtime. The per-point runtime is built through the **same** `loadLlmRuntimeConfigFromEnv` + `createLlmRuntime` factory as boot (cloud providers wrapped in `PhiGuardLlmRuntime`; `gemini-replit` ⇒ a fresh Gemini runtime), and cloud runtimes are cached per point. `summary` additionally honors the legacy `MEMORY_SUMMARY_MODEL` as its model id when `LLM_SUMMARY_MODEL` is unset (back-compat with M16). **Default-INERT:** a point with **no** `LLM_<POINT>_*` (and, for summary, no `MEMORY_SUMMARY_MODEL`) resolves to the global `getLlmRuntime()` + its prompt-pinned default model — byte-identical to the pre-M17 single-runtime path. The credential-free eval gate sets none of these, so it stays byte-identical. This is **static** per-point selection only; picking a model per *request* from the input (an LLM **router**) is a deferred future milestone (see `docs/MILESTONES.md`).
- Optional A2A cross-cloud mTLS + per-claim peer ABAC (threat model §Spoofing / §EoP — "A2A authorization"): `A2A_REQUIRE_MTLS=1` turns on the transport seam (inert otherwise — loopback dev has no PKI). Client side needs `A2A_MTLS_CERT` + `A2A_MTLS_KEY` (PEM paths) [+ `A2A_MTLS_CA`]; the `undici` dep is lazy-loaded (operators `pnpm --filter @workspace/api-server add undici`). Server side requires evidence of a verified client cert: direct TLS (`req.socket.authorized`) or an ingress verify header (`A2A_MTLS_CLIENT_VERIFY_HEADER` default `x-ssl-client-verify`, `A2A_MTLS_CLIENT_VERIFY_VALUE` default `SUCCESS`). **Per-claim ABAC against the peer's SAN/DN** layers on top, both controls opt-in: `A2A_MTLS_ALLOWED_PEERS` (CSV of allowed cert identities — matched against the peer's SAN entries [raw `URI:x` or bare `x`], CN, or full subject DN) refuses any non-allow-listed peer at the transport layer; `A2A_MTLS_PEER_BINDINGS` (CSV of `peerId=callerSubject` pairs) binds the network identity (cert) to the application identity (caller-identity JWT `sub`) so the two must agree (defeats a confused-deputy with a valid cert but a spoofed JWT subject). Peer identity is read from direct TLS (`getPeerCertificate`) or an ingress-forwarded URL-encoded client cert PEM (`A2A_MTLS_CLIENT_CERT_HEADER` default `x-ssl-client-cert`) / subject-DN header (`A2A_MTLS_SUBJECT_DN_HEADER` default `x-ssl-client-subject-dn`). All refusals 403 + ledger `a2a.transport_rejected` (system-scoped; payload = fixed route + static reason only — `no_verified_client_cert` / `peer_identity_unavailable` / `peer_not_allowed` / `peer_subject_mismatch`; never the peer DN/SAN or any header). With `A2A_REQUIRE_MTLS` unset, every one of these is inert and the eval gate stays byte-identical.
- Optional workflow/orchestration engine for the multi-agent supervisor: `WORKFLOW_ENGINE=inprocess|temporal` (default `inprocess` — the in-memory `InProcessWorkflowEngine` with CONCURRENCY=2 + CAS exactly-once, byte-identical to pre-seam behavior; no Temporal SDK loaded, eval gate byte-identical). The supervisor runs Triage → PHI-scan/ledger → Verifier → PHI-scan/ledger → persist on every `finding.created`; the engine only owns *how* that orchestration is scheduled. Same default-inert posture as `LOG_BUS_PROVIDER` / `RAW_EVIDENCE_PROVIDER`: nothing changes unless you opt in.
  - `temporal`: makes Temporal a real, durable backend. **Requires `TEMPORAL_ADDRESS`** (e.g. `my-namespace.acct.tmprl.cloud:7233` for Temporal Cloud, or `temporal:7233` for a self-hosted cluster) — boot fails fast if it's selected without an address (a misconfigured opt-in must never silently fall back to in-process). Optional: `TEMPORAL_NAMESPACE` (default `default`), `TEMPORAL_TASK_QUEUE` (default `phi-audit-review`), `TEMPORAL_TLS=1` (TLS to the cluster), `TEMPORAL_API_KEY` (Temporal Cloud API-key auth), `TEMPORAL_WORKFLOWS_PATH` (operator escape hatch to override the auto-resolved bundled workflow module path). The Temporal SDK is an **optional dependency** loaded lazily via a variable-aliased dynamic import (same as the cloud SDKs): operators must `pnpm --filter @workspace/api-server add @temporalio/client @temporalio/worker @temporalio/workflow` before enabling (a clear runtime error names the missing deps otherwise). Each `finding.created` starts a Temporal workflow whose id is a stable hash of `{tenantId, findingId}`, so a duplicate emit (e.g. double-emit on restart) is deduped by Temporal's start-workflow idempotency *in addition to* the DB CAS. The workflow is deterministic orchestration only — all I/O (DB CAS, LLM invoke, PHI scan, ledger, budget charge, persist) lives in activities registered with the worker — so a worker crash resumes from the last *completed* activity (replay reuses recorded results, it does not re-run them) instead of losing the job. **Activity auto-retry is enabled with bounded backoff (`maximumAttempts: 5`, exponential `1s→30s`):** transient step failures (LLM timeout, A2A blip, DB hiccup) self-heal instead of dropping the finding into `failed` for manual replay. This is safe because each side-effecting step is now per-step idempotent — the activity passes an `appendLedger` idempotency key derived from `workflowIdFor(tenantId, findingId)` + step name, which dedupes on the `ledger_idempotency_key_uniq` index (key NOT part of the ledger hash, so the chain stays byte-identical); a retried step that already wrote its `agent.*_complete` entry recovers the recorded verdict and returns without re-calling the LLM, re-charging the budget, or duplicating the audit row, and the per-tenant charge is ordered after the dedupe-gate write so it lands exactly once. Exactly-once-per-finding still also holds via the `acquireFinding` CAS + workflow-id idempotency. Per-tenant budget breaker, ledger event types, and PHI-before-ledger ordering are identical to the in-process path (the orchestration body is shared). **Worker topology:** N replicas poll the task queue (horizontal scale); the per-process budget breaker is per-worker (a cluster-wide budget would need a shared store — operator-deferred). **We do not stand up a Temporal cluster in Replit dev**; the adapter is unit-tested against a mocked client/worker.
- Optional event-bus transport: `LOG_BUS_PROVIDER=memory|kafka|nats` (default `memory` — in-process `InMemoryLogBus`, publish dispatches synchronously to the ingest pipeline; no broker, eval gate byte-identical). The bus carries `LogRecord`s on the `raw.logs` topic from log sources into the detector pipeline (ARCHITECTURE.md §3) — this is a *pre-detection* transport, so attacker-controlled raw `payload` bytes legitimately transit it; the relevant control here is broker transport security + access control (threat_model §"TLS everywhere"), exposed via the SSL/SASL (Kafka) and TLS/auth (NATS) options below. Redaction happens downstream in ingest. Records are JSON-encoded (Date↔ISO) and Zod-revalidated on the consume side (poison messages are logged + dropped, never crash the consumer). Delivery is at-least-once: a handler throw refuses the ack so the broker redelivers; re-processing is safe because ingest dedupes by fingerprint. **No `DEPLOYMENT_TARGET` shortcut** — moving log transport onto a real broker is always explicit (it needs broker addresses), like `SEARCH_PROVIDER`/`RAW_EVIDENCE_PROVIDER`. Broker SDKs (`kafkajs`, `nats`) are lazy-loaded inside `connect()`; operators must `pnpm --filter @workspace/api-server add <kafkajs|nats>` before enabling (a clear runtime error names the missing dep otherwise).
  - `kafka` (Kafka / Redpanda): requires `KAFKA_BROKERS` (CSV) [+ `KAFKA_CLIENT_ID` default `phi-audit`, `KAFKA_CONSUMER_GROUP` default `phi-audit-ingest`, `KAFKA_TOPIC` default `raw.logs`, `KAFKA_SSL=1` for TLS]. SASL is opt-in: setting `KAFKA_SASL_MECHANISM=plain|scram-sha-256|scram-sha-512` requires `KAFKA_SASL_USERNAME` + `KAFKA_SASL_PASSWORD`. Multiple replicas form one consumer group (horizontal scale); `tenantId` is the partition key so a tenant's records stay ordered.
  - `nats` (JetStream): requires `NATS_SERVERS` (CSV) [+ `NATS_STREAM` default `RAW_LOGS`, `NATS_SUBJECT` default `raw.logs`, `NATS_DURABLE` default `phi-audit-ingest`, `NATS_TLS=1`]. The stream is created if absent. Auth (first match wins): `NATS_TOKEN`, or `NATS_CREDS` (path to a `.creds` file), or `NATS_USERNAME` + `NATS_PASSWORD` (must be set together). A durable consumer with explicit/manual ack gives at-least-once (ack on success, nak → redeliver on handler failure).
- Optional cloud log source (M8 / M8.1): `LOG_SOURCE=cloudwatch|cloud_logging|azure_monitor` selects the pull adapter; unset ⇒ no source is started (dev fixture-replay only, unchanged). All three share the `PollingLogSource` loop (cursor in `log_source_checkpoints`, backoff, interruptible `stop()`, lifecycle ledger) and lazy-load their SDK — nothing is imported unless `LOG_SOURCE` selects that provider.
  - `cloudwatch`: `CLOUDWATCH_TENANT_ID` + `CLOUDWATCH_LOG_GROUPS` (CSV) + `AWS_REGION` [+ `CLOUDWATCH_POLL_INTERVAL_MS` (default 5000), `CLOUDWATCH_LOOKBACK_MS` (default 300000, only used when no checkpoint exists yet for a group), `CLOUDWATCH_MAX_CONCURRENT_GROUPS` (default 1 = sequential fetch, byte-identical to pre-M8.1; higher fans out log-group fetches with a bounded worker pool)]. Operators must `pnpm --filter @workspace/api-server add @aws-sdk/client-cloudwatch-logs`. Credentials follow the standard AWS chain (IRSA on EKS, env, instance role).
  - `cloud_logging` (GCP): `GCP_LOG_TENANT_ID` + `GCP_PROJECT_ID` [+ `GCP_LOG_FILTER` (Cloud Logging filter, default all entries), `GCP_LOG_POLL_INTERVAL_MS` (default 5000), `GCP_LOG_LOOKBACK_MS` (default 300000)]. Operators must `pnpm --filter @workspace/api-server add @google-cloud/logging`. Credentials follow ADC (Workload Identity on GKE, `GOOGLE_APPLICATION_CREDENTIALS`, etc.).
  - `azure_monitor`: `AZURE_LOG_TENANT_ID` + `AZURE_MONITOR_WORKSPACE_ID` + `AZURE_MONITOR_KQL` (the table/query, e.g. `AppTraces`) [+ `AZURE_LOG_POLL_INTERVAL_MS` (default 5000), `AZURE_LOG_LOOKBACK_MS` (default 300000)]. Operators must `pnpm --filter @workspace/api-server add @azure/monitor-query @azure/identity`. Credentials follow `DefaultAzureCredential` (Workload Identity on AKS, env, managed identity).
- Optional stuck-cursor reaper (M8.1): `INGEST_SOURCE_STALL_AFTER_MS` (switch — unset ⇒ disabled, nothing scheduled) [+ `INGEST_SOURCE_STALL_CHECK_INTERVAL_MS` (scan cadence, defaults to the stall threshold)]. When set, a leader-locked (own advisory key), `.unref()`ed periodic job scans `log_source_checkpoints` and emits a warning-level `ingest.source_stalled` ledger event for any cursor idle past the threshold so on-call learns a log source went quiet. **Edge-triggered**: alerts once per stall episode (in-memory latch keyed by source name), re-arms when the cursor advances again. Payload is source name + idle ms + threshold only (the checkpoint table holds no payload → no PHI). Default-INERT: unset ⇒ dev boot + the offline eval gate byte-identical (and with no cloud source configured the checkpoint table is empty anyway).
- Optional ingest dead-letter queue (M8.1): `INGEST_DEAD_LETTER_ENABLED=1|true` (switch — anything else ⇒ disabled) [+ `INGEST_DLQ_MAX_ATTEMPTS` (default 3, clamped 1..10), `INGEST_DLQ_BACKOFF_MS` (base delay, default 100, clamped 0..60000; doubles each retry, each computed delay capped at 60s)]. **DLQ off (default):** the pipeline processes each record once and RETHROWS on failure so a real broker no-acks + redelivers (at-least-once) and `/api/admin/ingest/replay` surfaces the error — byte-identical to pre-M8.1, the catch being a genuine poison record redelivers forever. **DLQ on:** the handler retries the record up to `INGEST_DLQ_MAX_ATTEMPTS` with exponential backoff, and on terminal failure persists a **metadata-only** marker to `ingest_dead_letter` (source pointer + `payload_sha256` + byte count + error *name* + attempt count — never the raw, possibly-PHI payload), ledgers `ingest.dead_lettered` (high), and ACKs so the broker stops redelivering the poison. The dead-letter write is best-effort — if it fails, the handler falls through to the original rethrow so nothing is silently lost. Default-INERT: unset ⇒ eval gate byte-identical.

## Embedder selection (cloud-aware)

The embedder used for hybrid retrieval is selected at boot via env. Precedence:
`EMBEDDING_PROVIDER` > `DEPLOYMENT_TARGET` > default (`featurehash`).

| Provider | Default model | Native HIPAA story | Required env |
|---|---|---|---|
| `featurehash` (dev) | — | n/a (local) | none |
| `bedrock` (AWS) | `amazon.titan-embed-text-v2:0` | BAA via AWS | `AWS_REGION` + SDK credential chain |
| `vertex` (GCP) | `text-embedding-005` | BAA via GCP | `GCP_PROJECT_ID`, `GCP_LOCATION`, ADC |
| `azure-openai` (Azure) | `text-embedding-3-small` | BAA via Azure | `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_DEPLOYMENT` |
| `tei` (cloud-agnostic) | server-side (e.g. `nomic-embed-text-v1.5`, `Qwen3-Embedding-0.6B`, `embeddinggemma-300m`) | data never leaves cluster | `TEI_ENDPOINT` |

`DEPLOYMENT_TARGET=aws|gcp|azure|local` is a shortcut that picks the cloud-native provider above; explicit `EMBEDDING_PROVIDER` always wins.

`EMBEDDING_DIM` (default `256`) sets both the embedder output dim and the `finding_embeddings.embedding` column dim. All four cloud providers' defaults support Matryoshka truncation to 256 natively. Changing dim on an existing DB: `DROP TABLE finding_embeddings; restart` — embeddings are a cache, backfill rebuilds them. The boot path validates that the existing column dim matches the configured one and fails loudly otherwise.

### Per-tenant pgvector partitioning (M12.2)

`EMBEDDINGS_TENANT_PARTITIONING` (default off; accepts `1|true|on|yes` / `0|false|off|no`/empty) selects the `finding_embeddings` storage layout at boot:

- **off (default)** — a single `finding_embeddings` table keyed by `finding_id`. Byte-identical to pre-M12.2; this is the dev + credential-free eval-gate path.
- **on** — `finding_embeddings` is `PARTITION BY LIST (tenant_id)` with a composite PK `(finding_id, tenant_id)`, an ivfflat index per partition, and a catch-all `finding_embeddings_default DEFAULT` partition so any un-provisioned tenant still stores + searches correctly. The conversion is idempotent: boot only drops + recreates the table when it exists and is not yet partitioned, then the embedding backfill rebuilds the rows (embeddings are a derived cache, so this is non-destructive).

Opt-in dedicated per-tenant partitions are provisioned with `provisionTenantEmbeddingPartition(tenantId)` (`lib/db/tenant-partition.ts`): it creates `finding_embeddings_<md5(tenantId)>` for an explicit `tenant_id`, is a no-op unless partitioning is on, and validates `tenant_id` against `SAFE_TENANT_ID`.

### Per-tenant KMS keys (M12.1)

There is **no env switch** — per-tenant signing keys are data-driven via the `tenant_kms_keys` control-plane table (`tenant_id` → `key_id` + `provider` + `key_ref`), loaded into an in-memory registry at boot. With an empty table (dev + eval-gate default) every resolve returns null and cookies sign/verify under the global `SESSION_SECRET`, byte-identical to pre-M12.1. Providers:

- `derived` — HKDF of the global `SESSION_SECRET` with the tenant as salt and `key_id:purpose` in info (distinct, non-colliding key per tenant/purpose; bump `key_id` to rotate). No extra dependency.
- `external` — `key_ref` names an **ENV VAR** holding the key material (operator wires Vault / cloud KMS → env). The table stores only the variable name, never the secret, so no key material lives at rest in the DB.

Actual cloud-KMS provisioning + rotation lifecycle is operator/Terraform territory; the application side is the table, the `resolveTenantSecret(tenantId, purpose)` resolver, and the auth-path wiring.
